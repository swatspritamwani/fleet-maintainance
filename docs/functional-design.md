## 3. DTO Validation Rules (Appendix B)

### `CreateRequestDto`

| Field | Type | Validation Rule | Error Message |
|-------|------|----------------|---------------|
| `vehicleId` | String | Required, not blank | "vehicleId: must not be blank" |
| `description` | String | Required, not blank, max 2000 chars | "description: must not exceed 2000 characters" |
| `priority` | Enum | Required, one of: LOW, MEDIUM, HIGH, CRITICAL | "priority: must be one of LOW, MEDIUM, HIGH, CRITICAL" |

### `AssignProviderDto`

| Field | Type | Validation Rule | Error Message |
|-------|------|----------------|---------------|
| `providerId` | UUID | Required, valid UUID format | "providerId: must be a valid UUID" |

Business rule (not DTO validation): provider must be active — returns 422 if inactive (AC-2.2).

### `InspectionReportDto`

| Field | Type | Validation Rule | Error Message |
|-------|------|----------------|---------------|
| `findings` | String | Required, not blank, max 5000 chars | "findings: must not be blank" / "findings: must not exceed 5000 characters" |
| `estimatedCost` | BigDecimal | Required, minimum 0.00 | "estimatedCost: must be greater than or equal to 0" (AC-3.1) |
| `estimatedDurationDays` | Integer | Required, minimum 1 | "estimatedDurationDays: must be greater than or equal to 1" (AC-3.2) |
| `attachments` | List\<String\> | Optional, each item a valid URI | "attachments[n]: must be a valid URL" |

### `DecisionDto`

| Field | Type | Validation Rule | Error Message |
|-------|------|----------------|---------------|
| `outcome` | Enum | Required, one of: APPROVED, REJECTED, INFO_REQUESTED | "outcome: must be one of APPROVED, REJECTED, INFO_REQUESTED" |
| `remarks` | String | Required when outcome = REJECTED or INFO_REQUESTED; max 2000 chars | "remarks: required when outcome is REJECTED or INFO_REQUESTED" (AC-4.2) |

Note: `remarks` conditional validation is implemented as a custom `@Constraint` annotation at class level on `DecisionDto`, not at field level.

## 4. State Machine Implementation Strategy

### Chosen Approach: Enum-based with method dispatch on the Aggregate Root

`MaintenanceRequest` owns all transition logic. Each transition is a method that:
1. Checks the idempotency condition (already in target state → return `this`).
2. Validates the guard condition (wrong source state → throw `IllegalStateTransitionException`).
3. Returns a new immutable instance with the updated state and a collected domain event.

```java
public enum Status {
    CREATED, ASSIGNED, INSPECTION_SUBMITTED,
    APPROVED, REJECTED, INFO_REQUESTED, PAYMENT_READY;

    public boolean isTerminal() {
        return this == REJECTED || this == PAYMENT_READY;
    }
}
```

### Transition Method Summary

| Method | Guard | Idempotency Condition | Emits Event |
|--------|-------|----------------------|-------------|
| `assign(providerId)` | status == CREATED | status == ASSIGNED && same providerId | `RequestAssignedEvent` |
| `submitInspection(report)` | status == ASSIGNED or INFO_REQUESTED | status == INSPECTION_SUBMITTED | `InspectionSubmittedEvent` |
| `approve()` | status == INSPECTION_SUBMITTED | status == APPROVED or PAYMENT_READY | `DecisionApprovedEvent` |
| `reject(remarks)` | status == INSPECTION_SUBMITTED | status == REJECTED | `DecisionRejectedEvent` |
| `requestInfo(remarks)` | status == INSPECTION_SUBMITTED | status == INFO_REQUESTED | `DecisionInfoRequestedEvent` |
| `transitionToPaymentReady()` | status == APPROVED | status == PAYMENT_READY | `PaymentReadinessEvent` |

### Why Not State Pattern?

The State Pattern (with a separate class per state) is more appropriate when each state has significantly different behaviour on the same operation. For this domain, the logic is simple guard checks and a linear transition table — enum-based is clearer, easier to test, and avoids unnecessary class proliferation.

## 5. Kafka Event Flow Diagram (per Use Case)

```
UC-1: Create Request
  Angular UI → POST /maintenance-requests
  → CreateMaintenanceRequestService.create()
  → MaintenanceRequest.create() [status: CREATED]
  → DynamoDB transactWrite: [Request row] + [Outbox: maintenance.request.created]
  → OutboxPublisher → Kafka: maintenance.request.created

UC-2: Assign Provider
  Angular UI → POST /maintenance-requests/{id}/assignments
  → AssignProviderService.assign()
  → MaintenanceRequest.assign(providerId) [status: ASSIGNED]
  → DynamoDB transactWrite: [Request row updated] + [Outbox: maintenance.request.assigned]
  → OutboxPublisher → Kafka: maintenance.request.assigned

UC-3: Submit Inspection
  Angular UI → POST /maintenance-requests/{id}/inspections
  → SubmitInspectionService.submit()
  → MaintenanceRequest.submitInspection(report) [status: INSPECTION_SUBMITTED]
  → DynamoDB transactWrite: [Request row] + [InspectionReport row] + [Outbox: maintenance.inspection.submitted]
  → OutboxPublisher → Kafka: maintenance.inspection.submitted

UC-4: Approve
  Angular UI → POST /maintenance-requests/{id}/decisions { outcome: APPROVED }
  → MakeDecisionService.decide()
  → MaintenanceRequest.approve() [status: APPROVED]
  → MaintenanceRequest.transitionToPaymentReady() [status: PAYMENT_READY]  ← automatic
  → DynamoDB transactWrite: [Request row] + [Decision row] + [Outbox: maintenance.decision.approved]
                                                            + [Outbox: maintenance.payment.ready]
  → OutboxPublisher → Kafka: maintenance.decision.approved
  → OutboxPublisher → Kafka: maintenance.payment.ready

UC-4: Reject
  → MakeDecisionService.decide()
  → MaintenanceRequest.reject(remarks) [status: REJECTED]
  → DynamoDB transactWrite: [Request row] + [Decision row] + [Outbox: maintenance.decision.rejected]
  → OutboxPublisher → Kafka: maintenance.decision.rejected

UC-4: Request Info
  → MakeDecisionService.decide()
  → MaintenanceRequest.requestInfo(remarks) [status: INFO_REQUESTED]
  → DynamoDB transactWrite: [Request row] + [Decision row] + [Outbox: maintenance.decision.info-requested]
  → OutboxPublisher → Kafka: maintenance.decision.info-requested

UC-5: Payment Ready (automatic on approve — see UC-4 Approve above)
  maintenance.payment.ready event payload includes: requestId, vehicleId, providerId,
  approvedCost (Money), estimatedDurationDays, approvedBy, approvedAt (§6.3)
```

### JWT Role Claim

Role is extracted from the JWT `roles` claim (provided by external IdP, §1.3):
- Claim name: `roles`
- Values: `COORDINATOR` or `SERVICE_PROVIDER`
- Angular: extracted in `AuthService.getRole()` and used by route guards and action button visibility logic.
- Backend: extracted via Spring Security `@PreAuthorize("hasRole('COORDINATOR')")` or checked in application service layer.
