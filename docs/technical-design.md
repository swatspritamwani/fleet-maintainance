## 2. DDD Package Layering

Enforced by GR-04 and ArchUnit tests. Dependency direction is strictly top-down:

```
controller  ──→  application  ──→  domain  ←──  infrastructure
                                     │
                                   port/
                               (interfaces only)
```

### Dependency Rules

| Layer | May Import From | Must NOT Import From |
|-------|----------------|---------------------|
| `controller/` | `application/`, Spring Web, generated API interfaces | `domain/`, `infrastructure/` |
| `application/` | `domain/` (models + ports), MapStruct mappers | Spring Web, AWS SDK, Kafka |
| `domain/` | Java standard library only | Spring, AWS SDK, Kafka, JPA |
| `infrastructure/` | `domain/port/`, AWS SDK, Spring Kafka | `controller/`, `application/service/` |

## 3. DynamoDB Single-Table Design

**Table name**: `fleet-maintenance`
**Billing mode**: PAY_PER_REQUEST

### Key Schema

| Entity | PK | SK |
|--------|----|----|
| `MaintenanceRequest` | `REQ#<requestId>` | `REQ#<requestId>` |
| `InspectionReport` | `REQ#<requestId>` | `INSP#<reportId>` |
| `Decision` | `REQ#<requestId>` | `DEC#<decisionId>` |

All entities for a given request share the same `PK`, enabling a single `query(PK = REQ#<id>)` to fetch the full request aggregate with all its children in one round trip.

### Global Secondary Index

**GSI name**: `status-createdAt-index`

| GSI Attribute | Mapped From |
|---------------|-------------|
| GSI PK | `STATUS` (string: e.g., `CREATED`) |
| GSI SK | `CREATED_AT` (ISO-8601 string) |

Used by the Request List screen (`GET /maintenance-requests?status=ASSIGNED`) to filter requests by status with recency ordering.

### Outbox Table

**Table name**: `fleet-maintenance-outbox`
**Billing mode**: PAY_PER_REQUEST

| Field | Type | Description |
|-------|------|-------------|
| `PK` | String | `OUTBOX#<eventId>` |
| `SK` | String | `OUTBOX#<eventId>` |
| `eventId` | String | UUID |
| `eventType` | String | e.g., `maintenance.request.created` |
| `kafkaTopic` | String | Target Kafka topic |
| `messageKey` | String | `requestId` |
| `payload` | String | Serialised `KafkaEventEnvelope` JSON |
| `status` | String | `PENDING` or `PUBLISHED` |
| `createdAt` | String | ISO-8601 UTC |
| `publishedAt` | String | Set on successful publish |
| `retryCount` | Number | Incremented on failed attempt |
| `TTL` | Number | Unix epoch — auto-expire PUBLISHED entries after 7 days |

**GSI**: `status-createdAt-index` (GSI PK=`status`, GSI SK=`createdAt`) — for efficient `PENDING` queries by the outbox polling publisher.

### Access Patterns

| Screen / Use Case | Access Pattern | DynamoDB Operation |
|-------------------|---------------|-------------------|
| Get request detail | By `requestId` | `query(PK = REQ#<id>)` — returns request + all inspections + all decisions |
| List requests by status | By `status` + date range | `query(GSI, PK = <status>, SK between dates)` |
| Submit inspection | Write new inspection | `transactWriteItems([Request update, InspectionReport put, OutboxEvent put])` |
| Make decision | Write decision + state change | `transactWriteItems([Request update, Decision put, OutboxEvent put(s)])` |
| Outbox poll | Fetch PENDING events | `query(GSI outbox, PK = PENDING, SK >= <lastPolled>)` |

## 4. Transactional Outbox Pattern — Implementation Decision

**Chosen approach**: Polling publisher with `@Scheduled`

**Rationale**: DynamoDB Streams + Lambda CDC would introduce Lambda cold starts and additional AWS infrastructure. For this use case, a polling publisher running within the BFF process on a 500ms interval is simpler, easier to test, and sufficient for the NFR-6 Kafka publish < 200ms p95 target (since the event reaches Kafka within the next poll cycle after the DB write, well under 1 second in steady state).

**Polling publisher interval**: 500ms (`@Scheduled(fixedDelay = 500)`)
**Batch size**: 50 events per poll cycle
**Retry strategy**: exponential backoff — retry counts 1–3: immediate; 4–6: 5s delay; 7+: dead-letter to separate DLQ topic `maintenance.events.dlq`

## 5. Container Image Build Strategy

### BFF Image (Multi-stage)

```
Stage 1 — Build:  eclipse-temurin:21-jdk-alpine
  mvn -B package -DskipTests
  Output: fleet-maintenance-bff.jar

Stage 2 — Runtime:  eclipse-temurin:21-jre-alpine
  Non-root user: fleet/fleet
  EXPOSE 8080
  ENTRYPOINT ["java",
    "-XX:+UseContainerSupport",
    "-XX:MaxRAMPercentage=75.0",
    "-Djava.security.egd=file:/dev/./urandom",
    "-jar", "app.jar"]
```

### Frontend Image (Multi-stage)

```
Stage 1 — Build:  node:20-alpine
  npm ci && ng build --configuration production
  Output: dist/fleet-maintenance-frontend/

Stage 2 — Serve:  nginx:1.25-alpine
  COPY dist/ → /usr/share/nginx/html
  Custom nginx.conf with try_files for Angular SPA routing
  EXPOSE 80
```

### Image Tagging Strategy

- CI builds: tagged with `git SHA` (`${{ github.sha }}`)
- Releases: additionally tagged with semver (`v1.2.3`) and `latest` for the default branch

## 6. Kubernetes Manifest Summary

| Manifest | Resource Type | Description |
|----------|---------------|-------------|

| `service-bff.yaml` | Service | ClusterIP, port 80 → 8080 |
| `service-frontend.yaml` | Service | ClusterIP (or LoadBalancer for external access), port 80 |
| `hpa-bff.yaml` | HorizontalPodAutoscaler | min=2, max=10, CPU target=70% |

### Health Probe Configuration

```yaml
livenessProbe:
  httpGet: { path: /actuator/health/liveness, port: 8080 }
  initialDelaySeconds: 30
  periodSeconds: 10
  failureThreshold: 3
readinessProbe:
  httpGet: { path: /actuator/health/readiness, port: 8080 }
  initialDelaySeconds: 20
  periodSeconds: 5
  failureThreshold: 3
```

Actuator groups configured in `application.yml`:
```yaml
management:
  endpoint.health.probes.enabled: true
  health.livenessState.enabled: true
  health.readinessState.enabled: true
```

**Quality gates that block merge**:
1. `lint` must pass
2. `build-and-test` must pass (includes JaCoCo 80% threshold and SonarQube Quality Gate)
3. `frontend-test` must pass
4. `static-analysis` must pass (zero Checkstyle violations, zero SpotBugs HIGH/CRITICAL, OWASP no HIGH CVEs)
5. `trivy-scan` must pass (no HIGH/CRITICAL CVEs with available fixes)

## 9. Environment Configuration Strategy

|---------|---------|-------------|
| `default` | Local developer run | No profile set |

### Configuration Files

```

├── application.yml               ← Shared defaults (no secrets)
├── application-test.yml          ← DynamoDB Local endpoint, embedded Kafka
```

### `application.yml` (shared defaults)

```yaml
spring:
  application.name: fleet-maintenance-bff
  kafka:
    bootstrap-servers: ${KAFKA_BOOTSTRAP_SERVERS:localhost:9092}
    producer:
      acks: all
      enable-idempotence: true

fleet:
  dynamodb:
    table-name: ${DYNAMODB_TABLE_NAME:fleet-maintenance}
    endpoint: ${DYNAMODB_ENDPOINT:}   # empty = use real AWS
    region: ${AWS_REGION:us-east-1}

management:
  endpoints.web.exposure.include: health,info,metrics,prometheus
  endpoint.health.probes.enabled: true

logging:
  pattern.console: '{"timestamp":"%d","level":"%p","service":"fleet-maintenance-bff","correlationId":"%X{correlationId}","message":"%m"}%n'
```

### Secret Injection (Production)

All secrets come from Kubernetes Secrets → environment variables:

| Secret Key | K8s Secret Name | Used As |
|-----------|----------------|---------|
| `aws-access-key-id` | `fleet-maintenance-secrets` | `AWS_ACCESS_KEY_ID` |
